# 新项目启动 + 需求共创 Prompt（从零开始 · 可直接复制发送）

你现在要在「AI 协作工作流 Starter Kit v3」这套机制下，协助我**从零启动一个新项目**。
**先不要写任何代码，也不要急着给完整方案。** 你的顺序是：理解机制 → 和我把「要做什么应用」聊清楚 → 把结论固化进项目状态文件 → 才进入开发。

---

## 0. 你必须遵守的机制（协作 OS · 最高优先）
1. **文件系统是唯一可靠状态源（SSOT）**：不靠会话记忆，所有结论最终落到项目文件。
2. **Hook / 自动检查 > 口头约束**；质量红线可量化（单文件≤800行 / 函数≤30行 / 嵌套≤3 / 分支≤3）。
3. **渐进式上下文加载 > 全量灌入**：按任务复杂度决定读取范围，别默认全读。
4. **双层分离**：协作 OS 层（本 kit，与技术栈无关）vs 应用架构层（我要做的产品 `src/`）。别混。
5. **收工按 tier 分级回写**（micro / feature / milestone），不过度仪式。
> 完整规则见仓库 `AGENTS.md` / `docs/00-context/AI_WORK_CONSTITUTION.md`，按需读取。

## 协作风格（对我）
- **分批提问**：每批 3–5 个高杠杆问题，等我回答再追问；不要一次抛 20 个，也不要替我擅自假设。
- 我答不上来的，给 2–3 个常见选项 + 你的推荐，帮我决策。
- 边聊边把「✅已确定 / ⏳待定」列出来让我确认。
- 简洁、说人话、有结论。

## 第 1 步：确认机制就位
- 检查项目根是否已含本 kit（关键文件：`AGENTS.md`、`.cloud.md`、`docs/00-context/PROJECT_MAP.md`、`scripts/checks/`、`.githooks/`）。
  - 已含：读 `.cloud.md` 与 `PROJECT_MAP.md`（此时多为空模板）。
  - 未含：提醒我「先把 Starter Kit 复制到项目根」，并给启用命令：
    `git config core.hooksPath .githooks && chmod +x .githooks/* scripts/checks/* scripts/*.sh`
- 用 1–2 句话说明你已在机制下待命，然后进入第 2 步。

## 第 2 步：需求共创（核心 · 和我对话，别自问自答）
目的：把「我脑子里的模糊想法」变成「可开发的清晰定义」。按下面 7 个维度，分批问我：

1. **问题与价值**：解决谁的什么问题？不做会怎样？一句话价值主张。
2. **目标用户与场景**：主要用户是谁？最高频的 1–2 个使用场景？
3. **核心功能（MVP）**：第一版**必须有**哪些？明确**先不做**哪些（防范围蔓延）。
4. **约束**：技术栈倾向（默认 Next.js / React / TS，可改）、UI/组件库、性能/安全/合规、时间与人力。
5. **数据与集成**：核心数据实体？要对接的外部系统 / API / 支付 / 登录？
6. **成功标准**：第一个里程碑「做成什么样算成功」（可验收口径）。
7. **风险 / 未知**：最大的不确定点、依赖、可能的坑。

## 第 3 步：输出理解并与我确认（≤8 条，等我点头）
- 应用一句话定义 / 目标用户 / 核心价值
- MVP 范围（做什么）+ 明确非目标（不做什么）
- 关键约束（技术栈 / 组件库 / 质量红线）
- 第一里程碑的验收标准
- 建议的应用架构骨架（模块划分 + 依赖方向，见第 4 步）
- 主要风险 / 待定项
- 建议 tier 与首个迭代范围
> **未经我确认，不要进入第 4 步。**

## 第 4 步：固化进项目文件（把对话变成 SSOT · 先给草稿，我同意再落盘）
- `docs/00-context/PROJECT_MAP.md`：业务目标 / 约束 / **模块地图** / **依赖规则** / 数据与接口 / 文档索引。
- `.cloud.md`（含 frontmatter）：`goal`、`tier`、`scope_in/out`、`next_actions`、`active_session`、`updated`。
- `docs/20-plan/current-iteration.md`：第一迭代目标 + checklist（每项带验收标准）。
- `docs/10-spec/<feature>.md`（首功能值得写规格时，用 `spec-template`）。
- `ai/sessions/<date>--kickoff.md`（用 `_session-template`，并登记 `ai/sessions/INDEX.md`）。

### 应用架构骨架（应用层 · 按聊清楚的类型给）
给出推荐的 `src/` 结构与**单向依赖方向**（**feature 模块化单体**起步，避免过早微服务）：
- 分层：`app/`(路由/页面) → `features/*`(按业务能力切) → `domain/`(纯业务逻辑) → `shared/`(通用)；`infra/`(DB/外部API) 仅经接口被注入。
- 依赖方向：`app → features → domain → shared`；`infra` 不被 domain 反向依赖；`features` 之间不直接互相 import。
- 把边界写进 `PROJECT_MAP` 依赖规则 + `quality/import-boundaries.txt`（启用 opt-in 边界校验）。
- 列出首批关键文件（模块入口 / 服务层 / 适配层），它们需有头部说明块（`input/output/pos`），登记到 `quality/key-files.txt`。

## 第 5 步：切第一个最小纵切并开工
- 从 MVP 挑一个**端到端最小纵切**（能跑通的一条主线，而非横向搭半成品）。
- 按 **Start → Plan → Execute → Verify → Sync → Close** 推进；每完成一项立即回写 checklist。
- 收工按 tier 用「收尾落盘」（Claude Code 直接 `/close`）。
- Hook / 检查失败不得强推；超质量红线不得静默（记 session 或 ADR）。
