# Meme Daily — 产品方案文档

## Context（为什么做、做成什么样）

**Meme Daily** 是一个**公司内部**的每日中文网络热梗参考工具，服务对象是**公司传播 / 内容 / 营销运营团队**，定位为他们的**选题参考与传播咨询台**——不是面向大众的内容站。由 Codex 每天早上自动从微博、抖音、小红书、B站抓取候选，挑出约 10 个**对内容创作 / 广告营销 / 短视频选题有参考价值的热梗**（覆盖热点事件梗、短视频梗、生活方式梗、二创梗），每个梗说明**来源 / 典型用法与传播场景 / 有趣点 / 为什么被传播放大 / 还能不能上车 / 怎么用到品牌**，每天 8 点自动推送到页面，并按日期永久存档 + 可跨天检索。

核心立意是**“不是单纯的新闻热度，而是有复用与营销价值的梗”**。

**定位带来的范围裁剪（本轮已定）**：因为是内部工具——
- **不做对外分发**（邮件 / RSS / SEO / 机器人推送）、**不做商业化**；
- **抓取交给 Codex 负责，不在本文档展开**。

于是产品难点收敛为两件事：①判断一个梗值不值得发、怎么用的编辑逻辑（护城河）②全自动发布不翻车的安全闸；外加一套给专业运营每天用的高密度 UX。

本文档沉淀已对齐的设计决策，作为后续落地的依据。

---

## 一、总体架构：大脑 / 哑终端分工

- **Codex = 大脑**：负责抓取 → 安全过滤 → 选梗判断 → 写卡片 → 自检 → 推送。所有复杂度和风险集中在这一侧。
- **静态站 = 哑终端**：只做“读 json → 渲染卡片 → 按平台/类型筛选”，不含业务逻辑。
- **传输方式（已定）= git commit**：Codex 每天把一个 json 文件提交到仓库，CI（Vercel/Netlify/GH Actions）检测改动自动重建站点。
  - 版本历史 = 免费的审计日志；紧急下架 = 一次 commit；天然幂等；零额外基础设施。

## 二、推送数据契约（产品的“宪法”）

这份契约同时承担三个角色：**渲染格式 + 质量底线 + 紧急下架开关**。

### 文件布局：一天一个文件，目录即归档
```
data/daily/2026-06-20.json
data/daily/2026-06-19.json
...
```
站点构建时直接 glob 该目录生成归档列表，**不单独维护 index**（单一数据源，永不漂移）。同一天重跑覆盖同名文件，天然幂等、可纠错。

### 信封（每天一份，兼作“每日回执”）
```json
{
  "schema_version": "1.0",
  "policy_version": "1.0",
  "rubric_version": "1.0",
  "date": "2026-06-20",
  "generated_at": "2026-06-20T07:42:00+08:00",
  "status": "published",            // published | partial | skipped | held
  "run_report": {
    "candidates_scanned": 37,
    "published": 7,
    "dropped_safety": { "政治时政": 2, "社会事件": 1, "明星争议": 3, "未成年人": 1, "隐私": 0, "辱骂攻击": 1 },
    "dropped_low_confidence": 2,
    "sources": ["weibo", "douyin", "bilibili"]
  },
  "items": [ ... ]
}
```

### 条目（MVP 瘦描述层）
```json
{
  "id": "2026-06-20-someSlug",       // 稳定 id = 永久链接 + 去重 + 连续热点关联
  "title": "梗名",
  "aliases": ["别名/变体写法"],
  "platform": ["douyin", "weibo"],   // 封闭枚举
  "type": "短视频梗",                // 封闭枚举：热点事件梗 | 短视频梗 | 生活方式梗 | 二创梗
  "summary": "一句话是什么",
  "origin": "来源/出处（必须来自当天抓取，不许凭记忆编）",
  "usage": "典型用法 / 传播场景",
  "fun_point": "有趣点",
  "why_spread": "为什么被传播放大",
  "sources": [                       // ≥1 条；反幻觉的锚；缺了就不发这条
    { "platform": "douyin", "url": "https://...", "captured_at": "2026-06-20T07:10:00+08:00" }
  ]
}
```

### 合同条款（契约的灵魂）
1. **必填脊柱**（站点强依赖、以后基本不动）：信封 `schema_version / date / status / items[]`；每条 `id / title / platform / type / summary / origin / usage / fun_point / why_spread / sources(≥1)`。
2. **MVP 就产出的核心增强字段**（仍按“有则渲染”实现，但内容上 MVP 即产出）：`lifecycle`（rising|peak|declining，“还能上车”）/ `brand_usage`（品牌借用示范）/ `risk`（软风险提示）。**原因**：定位转为内部营销参考后，“还能上车 / 怎么用 / 安不安全”才是用户真正来拿的核心价值，不是可选增强（详见第六节）。
3. **可继续延后的增强字段**：`score` / `confidence` / `days_on_list` / `tags` / `aliases`。
4. **铁律——新增可选字段永远不算破坏性变更**；只有改名/删除/改义一个必填字段才算（前者升 `schema_version` 小版本，后者升大版本）。站点对缺失的可选字段一律“不渲染”，于是旧归档永不因加字段而崩。
5. **封闭枚举**：`platform / type / lifecycle / risk.level / confidence / status` 只能取约定值；站点对集合外的值要有“未知”兜底桶，不许崩。
6. **每条 ≥1 个信源**，否则 Codex 自己丢弃、不发（反幻觉硬约束）。
7. **校验是渲染的前置闸**（见下）。

### 契约顺手办掉的三件事
- **紧急下架**：信封 `status: "held"`（或某条 `published: false`），commit 即隐藏；手机上改一个字段 ~20 秒。
- **去重 / 连续热点**：Codex 写今天前先读最近 N 天 json，算 `days_on_list`、避免重发——**目录既是输出、也是 Codex 的记忆**，不用额外数据库就有了状态。
- **幂等纠错**：同日期 = 同文件 = 覆盖；git 历史保留旧版。

### 关键约束：安全 ≠ 渲染字段
**安全过滤、丢低置信、去重都是“管线行为”（Codex 直接不吐这条），不是渲染字段。** 即使 MVP 很瘦，这些动作照常发生，只是暂不在卡片上显示徽章。脊柱 = 渲染出来的部分；管线背地里干的活比脊柱多。

## 三、选梗 Rubric（护城河：选什么）

### 梗 vs 新闻的本质判据：有没有一个可填的“壳”
- **梗 = 有可复用的壳**（句式 / BGM / 视觉模板 / 动作表情），能脱离原事件套到新语境 → 有二创、有传播、品牌可借用。
- **新闻 = 一次性事件**，热度高但没壳、套不到别处、品牌碰不得。
- 第一性问题不是“火不火”，而是“**有没有能复用的壳**”。

### 最关键结构：两段式 = 硬门槛（一票否决）+ 软打分（排序）

**第一段·硬门槛（任意一条不过 = 出局，不进打分）**

| 门槛 | 问题 | 不过就丢 |
|---|---|---|
| 是不是梗 | 有可复用的壳？能套新语境吗？ | 纯新闻/事件 |
| 真实传播 | 真用户二创还是营销号刷的？跨几个平台？ | 刷量伪梗 |
| 品牌安全 | 见第四节禁发清单 | 敏感内容 |
| 置信度 | 证据够不够支撑判断？ | 拿不准的 |

**第二段·软打分（过门槛者排序，取过线 top ≤10）**

| 维度 | 问题 | 倾向 |
|---|---|---|
| 套用潜力 | 壳多通用？能套多少品牌语境？ | 最高权重 |
| 时机/生命周期 | 刚起势 / 正热 / 过气？ | **起势分最高**（营销最想要“还能上车”） |
| 创作落地性 | 品牌能否 48h 内做出一条内容？怎么做？ | to-B 价值 |
| 受众宽度 | 大众秒懂 vs 窄圈黑话？ | 越宽越好 |

### 铁律：每个判断都要“拿证据”，不许凭 vibes
- “套用潜力高” → 必须给 ≥2 个不同语境的二创实例。
- “真实传播” → 必须给跨平台 / UGC 数量证据。
- 给不出证据 = 编 = 不发。`score` 与 `sources` 天然绑定，分数可审计。

### Prompt = 分步管线（多趟 > 单趟）
1. **抽候选**：从热榜+详情列出原始候选 + 证据片段，先不判断。
2. **过硬门槛**：逐个 pass/drop + 理由（保守，拿不准就 drop）。
3. **打分排序**：幸存者按软维度打分，每维度强制带证据，取 top。
4. **写卡片**：只给入选者写描述层字段，严格基于证据。
5. **自检（单独一趟、扮演杠精）**：换怀疑者视角回审“出处是不是编的 / 信源真存在吗 / 这条其实是新闻吧 / 分虚高吗”，没过的降级或丢弃。

### 因为无人日审，需两样兜住“判断漂移”
- **few-shot 金标准集**：手挑 3–5 个“好梗（高分+理由）”和 3–5 个“伪梗/新闻（出局+理由）”进 prompt，随时间养大。
- **每周调参回路**：每周 10 分钟扫临界案例（差点出局/差点入选），微调阈值与 few-shot。rubric 是活的。
- **工程技巧**：判断趟用低温度；带 `rubric_version`；全程结构化输出（每维度分数+证据+裁决）便于审计调参。

## 四、安全闸（护城河：毙什么）—— 全自动的保命线

### 禁发清单（命中任意一类 = 直接丢弃，不进价值打分）

| 类别 | 覆盖 |
|---|---|
| **政治时政**（必须，最高优先级） | 时政敏感——漏一个可能直接关停 |
| 社会事件 | 事故、灾害、犯罪、公共悲剧、敏感社会议题 |
| 明星争议 | 塌房、绯闻、对立、私德争议（封“争议”，非所有明星内容） |
| 未成年人 | 以未成年人为主体/可识别的内容 |
| 隐私 | 人肉、私人信息、素人被卷入 |
| 辱骂攻击 | 网暴、人身攻击、对线、地图炮、歧视 |
| （建议同补） | 色情低俗擦边、违法血腥暴力、可能致害的谣言 |

> 用户最初列了后五类；**政治时政等是必须补上的存在性风险项**。

### 三条设计原则
1. **独立一趟，不和价值打分混**——安全是冷判断、一票否决，正交于价值。
2. **默认拒绝（拿不准就丢）**——误杀好梗的代价远小于放过坏内容；提示词写死这个不对称。
3. **drop 是管线行为**——被毙内容不进 `items[]`。

### 两层防御（关键词规避是常态）
- **第一层·关键词/规则预筛**：维护敏感词/人名/事件词表，命中即在喂 LLM 前丢。
- **第二层·LLM 语义分类**：抓谐音/缩写/拼音/emoji/暗示等规避，逐类别出“裁决+理由+证据”。
- 两层都“命中即丢”：一层防呆、二层防绕。

### 边缘难题的默认策略
- **梗干净但出处脏**（如源自明星塌房的流行句式）：不引用脏出处就讲不清 → 丢；只有完全脱离原事件、壳能独立复用才考虑放行，且“出处脏+还很新”一律先丢。
- **明星**：全自动下，凡以某真实个人为主体的一律降级/丢，躲隐私与诽谤。
- **未成年人**：以未成年人为主体的，主体即丢。

### 落进契约
- **带版本号的发布策略文档（`policy_version`）**，安全趟照它判。
- **`run_report.dropped_safety` 按类别拆**，进每日回执——每天 30 秒看“因什么、丢了多少”，反向校准词表与策略。

### 责任提醒
全自动发布中文热榜衍生内容，运营方对内容负审核责任（政治类尤甚）。本方案“语义安全闸 + 紧急下架 + 每日回执扫一眼”是**负责任全自动的最低配**（事前自动拦 + 事后随时撤 + 每天扫一眼），不是裸跑。产品做大或商业化后，监管可能要求真人内容审核，需提前有数。这与“全自动”不冲突：仍不需早上盯着发布或逐条点确认。

## 五、边界与降级策略
- **抓不到怎么办**：预定义降级——发更少 / 跳过当天（`status: skipped`）/ 复用昨天仍在热的；绝不让失败推出垃圾或搞崩站点。
- **不凑满 10 条**：某天只有 6 条过线就发 6 条；硬凑会用弱梗稀释品牌。规则=“最多 10、只发过线的”。
- **时区钉死**：`date`、cron、抓取统一 `Asia/Shanghai`，避免日期差一天。
- **媒体不进仓库**：只存信源 URL，躲版权 + 防仓库膨胀。
- **一次 commit = 一个完整的天**：内存里拼好整份 → 验过 → 原子写入提交，绝不逐条增量提交。

## 六、页面与 UX（内部选题参考台）

### 受众与核心诉求（JTBD）
公司传播 / 内容 / 营销运营，专业用户、每天来。三个诉求：
- **每天 triage**：今早有什么值得蹭的梗？哪些还来得及上车？
- **决策 decide**：这个梗怎么用到品牌？安不安全？
- **查档 lookup**：上次那个梗哪天的？最近某类梗有哪些？

### 第一原则：信息密度 + 三秒决策（不是视觉冲击）
卡片正面只放决策必需的几样（梗名 + 一句话 + 平台/类型 + 生命周期徽章 + 风险标 + 价值分），其余折进详情。每条要支持三秒判断“关我事吗 / 还来得及吗 / 安不安全”。

### 四个界面
1. **今日台（主界面）**：日期 + **运行状态条直接并进页面顶部**（今日发几条 / 因什么毙几条——监控即产品，不另搭机器人）+ 筛选（平台/类型）与排序（默认“还能上车”）+ 紧凑卡片列表；**就地展开**看详情；**过气梗保留但视觉降权**（传递“别追了”本身有参考价值）。
2. **单梗详情（展开态）**：全字段摊开；`brand_usage` 高亮成“来拿的”块、`risk` 高亮成“要避的”块；信源带“@账号 + 抓取日期”可点。
3. **梗库检索**：「按天 / 梗库」两视图；梗库是跨天聚合的可检索库（搜梗名 + 别名 + 内容，按类型/平台/时间/排序筛）；**同一梗多天合并成一条**并显示“连热 N 天 / 出现 M 次”，出现次数即热度与生命周期信号；紧凑列表、与今日台共用视觉语言。
4. **单梗独立详情页（永久 URL）**：可在群里甩链接 / 复制带走；顶部就是分享与复制操作；字段全摊开；出现历史小柱状看热度趋势；相关梗横向跳转便于做“系列选题”。

### 内部工具特有取舍
- **免营销落地页 / 免注册**：放内网或公司 SSO 后即可。
- **监控并进产品**：每日回执做成今日台顶部状态条，不另推送（已定不做对外分发）。
- **复制带走 / 永久链接并存**：方便丢进选题文档、群里分享——内部协作高频动作。
- **就地展开 + 独立 URL 并存**：扫得快与可分享两者都要。
- **响应式即可**：主力工位电脑，不做独立 App。

## 七、分阶段建议
- **MVP**：Codex 抓取（交给 Codex，不展开）→ 安全闸（含政治时政）→ 两段式选梗 → 写描述层 + **核心增强（lifecycle / brand_usage / risk）** → 自检 → 校验 → git commit → 站点：**今日台**（卡片列表 + 平台/类型筛 + “还能上车”排序 + 顶部运行状态条）+ **就地展开详情** + **单梗独立详情页（永久 URL）**。
- **后续增强**：**梗库跨天检索**（搜索 + 同梗多天合并 + 出现次数）、`score` 精细化、出现历史柱状、相关梗推荐、`tags`、复制带走的多格式。
- **明确不做**（定位裁剪）：对外分发（邮件 / RSS / SEO）、商业化、登录墙式公开站。

## 八、校验 / 验收（实现后怎么判断真的跑通）
1. **契约层**：用 JSON Schema 验当天 json；构造一个缺 `sources` 的条目，确认被拒；构造非法枚举，确认站点走“未知”兜底不崩。
2. **校验闸**：故意推一份非法 json，确认 CI 构建失败但**线上保留上一次成功版本**。
3. **选梗**：拿一组含“真梗 + 纯新闻 + 伪梗（刷量）”的样本跑管线，确认新闻/伪梗被硬门槛挡下、真梗入选且每维度带证据。
4. **安全闸**：构造含政治时政/明星争议/未成年人/隐私/辱骂的样本，确认全部被丢且 `run_report.dropped_safety` 计数正确。
5. **紧急下架**：对某天置 `status: held` 并 commit，确认站点 1 次重建后隐藏当天。
6. **去重**：连续两天同一个梗，确认第二天标 `days_on_list` 而非重发新卡片。
7. **降级**：模拟抓取失败，确认产出 `status: skipped` 且站点不崩。
8. **端到端**：完整跑一个早晨，确认 8 点前站点出现当日页 + 日期永久链接 + 顶部运行状态条数据正确。
9. **UX / triage**：让一名运营对着今日台做一次真实选题，确认能在一两分钟内判断“哪些还能上车、哪个怎么用、有没有风险”。

## 九、开放问题与已决事项

**已决（本轮对齐）**：
